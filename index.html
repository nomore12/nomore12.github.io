database

예전에 장고에서 페이지네이션 구현할 때는 Django Restframework가 알아서 페이지네이션을 해줬어서 그 고마움을 몰랐다. 근데 생각해보니 pagenation 방식이 PageNumberPagination, LimitOffsetPagination을 사용했었다. 그래서 sql에서 limit랑 offset 구문을 이용해서 페이지네이션 하는거겠군 했다.

postgresql

Django에서 Postgresql이 좋다길래 테스트를 해봤다. “PostgreSQL로 시작하는 SQL코딩 입문 기본편, 활용편”을 도서관에서 빌려서 기본편을 보면서 했다. 일단 책 제목은 “SQL 코딩입문”인데 절대 SQL 코딩 입문이 아니다. db를 아주 잘쓴다고는 못해도 검색해가면서 어찌어찌 sql문도 만들고, django 쓰면서 orm도 써보고 해서 그나마 보는데 많이 어렵지는 않았는데(기본편 한정) 입문자가 보면 포기 할 것 같다.

어쨌든, postgresql 커맨드라인에서 연습용 DB를 만들고 pdAdmin4와 Dbeaver로 연결해봤는데 MySql과 뭔가 좀 다르다. 나는 분명 Database를 만들었다고 생각했는데 이름만 데이터베이스고 연습용 데이터베이스 안의 스키마가 데이터베이스 역할을 하고 있었다. 스키마에서 테이블을 만들고 컬럼을 정의하고 해당 스키마에서 sql 질의를 작성해야한다.

Gunicorn 가동

위 명령어로 gunicorn, unix socket으로 서버를 가동한다. 기능 테스트를 해도 잘 통과한다. 그리고 이유는 모르겟는데 유닉스 소켓으로 교체하라고 한다.

두 서버가 동시에 포트 8000을 사용 할 수 없다. 서로 다른 포트를 사용하도록 설장할 수 있지만, 이 방식은 약간 자의적이다.

또한 쉽게 스테이징 서버를 실제 운영 포트에서 가동한다든가, 그 반대의 경우도 발생 할 수 있어서 위험하다고 할 수 있다.

더 나은 해결책은 유닉스의 도메인 소켓을 이용하는 것이다. 이것은 디스크상에 파일 형태로 존재하며, nginx와 gunicorn이 이 파일을 이용하여 서로 커뮤니케이션할 수 있다.
