기능 테스트를 이용한 DJango 설치

  1. 가상환경을 만든다.
  2. Django와 Selenium을 설치한다.
  3. functional_test.py를 만든다.

책에는 AssertionError가 발생해야 한다고 하는데 나는 ERR_CONNECTION_REFUSED만 발생했다. 검색해보니 클린코드를 위한 테스트 주도 개발의 영문 버전이 있는 사이트가 있었고, 영문 예제에선 비슷한 에러가 발생했다. 책에 있는 예제처럼 assert까지 가지도 못하고 browser.get()에서 에러가 발생하는 것도 같았다. 대부분 비슷한 에러인데 마지막 ERR_CONNECTION_REFUSED는 다른 메시지가 출력되었다. 아무래도 책은 2014년판, 영문 사이트는 2017년이 마지막, 그리고 지금은 2020년이니 그동안 셀레니움 버전이 많이 올라가서 책과 다르게 출력되었던것 같다.

장고를 python manage.py runserver 명령으로 가동하고나면 functional_test에서 에러가 발생하지 않는다.

unittest 모듈을 이용한 기능 테스트 확장

셀레니움을 이용한 테스트에선 실제 웹 브라우저를 실행해서 어플리케이션이 어떻게 동작(functions)하는지 사용자 관점에서 확인했다. 이런 테스트를 기능테스트(functional test)라 한다. 또한 각각의 기능 테스트는 사용자 스토리 방식으로 표현할 수 있다.

예상했던 결과가 나온다. title은 To-Do가 아니어서 AssertionError가 출력된다. 이것은 테스트 실패에 대한 정당한 이유를 가지고 있기 때문에 테스트를 제대로 작성했다고 생각하면 된덴다.

unittest 모듈

앞의 AssertionError라는 메시지가 도움이 되지 않는다고 한다. 테스트가 브라우저 타이틀에서 어떤 문구를 찾았는지 알려준다면 유용할 것이고, 테스트가 끝나면 자동으로 닫히면 더 좋을듯.

브라우저의 타이틀명을 표시려면 assert 명령의 두 번째 인수에 넣으면 된다.

assert "To-Do" in browser.title, f"Browser title was {browser.title}"

책에는 문법이 좀 다른데 파이썬 버전 차이인듯 하다.

  1. (1.) unittest.TestCase를 상속해서 테스트를 클래스 형태로 만든다.
  2. (4.) 테스트의 메인 코드. test라는 명칭으로 시작하는 모든 메서드는 테스트 메서드이며 클래스당 하나 이상의 메서드를 작성 가능하다. 가능한 내용을 알 수 있는 테스트 메소드 명칭을 사용하는 것이 좋다.
  3. (2, 3) setUp과 tearDown은 각 테스트 시작과 끝에 실행된다. 브라우저를 시작하고 닫을 때 사용한다. try/except와 비슷한 그조로 테스트에 에러가 발생해도 tearDown이 실행된다.(setUp내에 exception이 있는 경우에는 tearDown이 실행되지 않는다.)
  4. (5.) test assertion을 만들기 위해 assert 대신 self.assertIn을 사용한다. unittest는 test assertion을 만들기위해 이런 유용한 함수를 다수 제공한다. assertEqual, assertTrue, assertFalse등.
  5. (6.) self.fail은 강제적으로 테스트 실패를 발생시켜서 에러를 출력한다. 위의 코드에선 테스트가 끝났다는 것을 알리기 위해 사용하고있다.

단위테스트를 이용한 간단한 홈페이지 테스트

단위테스트는 무엇이고 기능테스트와 어떤 차이가 있는가?

  • 기능 테스트는 사용자 관점에서 어플리케이션 외부를 테스트 하는 것
  • 단위 테스트는 프로그래머 관점에서 그 내부를 테스트 하는 것

TDD 접근법은 양쪽 테스트 모두를 적용하며, 이후 작업 순서는 다음과 같다.

  1. 기능 테스트를 작성해서 사용자 관점의 새로운 기능성 정의
  2. 기능 테스트가 실패하면 어떻게 코드를 작성해야 테스트를 통과할지 생각. 이 시점에서 하나 또는 이 이상의 단위 테스트를 이용해서 어떻게 코드가 동작해야 하는지 정의한다(기본적으로 모든 코드가 적어도 하나 이상의 단위 테스트에 의해 테스트 되어야 함)
  3. 단위 테스트가 실패하고 나면 단위 테스트를 통과할 수 있을 정도의 최소한의 코드만 작성하. 기능 테스트가 완전해질 때까지 2와 3을 반복해야 할 수도 있다.

이 과정을 잘 보면 기능 테스트는 상위 레벨의 개발을 주도하고, 단위 테스트는 하위 레벨의 개발을 주도하는 것을 알 수 있다.

Django에서의 단위 테스트

위의 테스트는 잘 동작한다(실패하는 쪽으로)

위 테스트를 실행하면 ImportError: cannot import name 'home_page' 라는 에러가 발생한다. home_page는 아직 존재하지 않기 때문이다.

위 테스트를 실행하면 url 패턴을 처리할 수 없다는 에러가 발생한다. 위에 작성한 기능 테스트를 통과 할 때까지 최소한의 구현을 하고, 실패하는 단위 테스트를 통과하게끔 다시 구현하는 과정을 통해 기능 테스트를 통과한다.

코드 품질을 높이기 위해선 코드 변경을 최소화 해야한다. 또한 이렇게 최소화한 코드는 하나하나 테스트에 의해 검증받아야 한다. 익숙해지기 시작하면 속도는 빨라진다고 한다. 이전까지 그랬왔던 것처럼 함수 하나를 다 만들고 실행해보는 방식은 지양해야겠다.

그리고 오래된 책이라 그런지 예제를 그대로 따라할 수 없었다. 아무 생각없이 따라 치기만 해선 되질 않으니 오히려 더 생각하면서 따라가게 되었다(그래서 시간이 더 오래걸린다…)
