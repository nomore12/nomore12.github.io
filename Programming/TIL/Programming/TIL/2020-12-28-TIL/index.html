Djgango blog setting

‘클린 코드를 위한 테스트 주도개발’ 6장 중반까지의 내용을 복습할겸 **’Tow Scoops of Django 3장 어떻게 장고 프로젝트를 구성할 것인가’**를 적용한 프로젝트를 만들기로 했다.

간단하게 프로젝트 루트 폴더를 만들고, 그 안에 가상환경, git, django 프로젝트를 생성했다.

settings 분리

책의 예제 프로젝트 구성처럼 config 폴더를 생성하여 settings.py를 옮기고 분리했다.

base.py에는 settings.py의 내용을 그대로 넣고, development.py와 produnction.py에 각각의 Database 설정만 다르게 설정했다. development.py와 prodjunction.py는 from .base impot *으로 base.py의 모든 셋팅을 적용되도록 임포트 해놓았다.

예제에는 Makefile이라는게 포함되어 있었는데, 개념을 잘 모르겠어서 일단 뺐다. 그리고 3장 끝날때까지 장고 프로젝트 템플릿만 설명하고 있어서 일단 runserver를 해봤다. 그랬더니 마구 에러를 뿜으며 죽는다. ‘아니 지은이 이 사기꾼들 왜 3장만 따라하면 되는것 처럼 해놓고 왜 안되는거야, 장고 버전이 올라가면서 문제가 생겼나’ 등의 생각을 했으나, settings.py를 다른 폴더로 옮기니까 당연하게 문제가 생길거라 생각했다. 그리고 검색을 통해 솔루션을 찾았다.

일단 base.py, asgi.py, wsgi.py, manage.py의 "blog.settings" 처럼 기존 settings.py를 이용하는 설정들을 "blog.config.base"로 바꿨다. (장고에서는 문자열로 폴더 경로를 읽어올수도 있다. 물론 os 모듈을 이용해서 사용할 수도 있다.)

그리고 runserver를 돌리면 서버는 잘 뜬다. 하지만 migrate에서 문제가 발생한다. development.py와 produntion.py로 분리하면서 base.py의 데이터베이스 설정을 날려버렸기 때문에 migration 파일들을 어디에 migrate 해야 하는지 모르는것 이었다.

이것을 해결하기 위해선 셋팅을 아래와 같이 환경변수를 입력해야 했다. 디폴트로 사용되는 셋팅은 blog 폴더안의 settings였는데, 이것을 config로 옮기고 DB정보를 나누어놨으니 blog.config.development로 읽으라고 해야하는 것이다.

development.py에는 base.py가 임포트 되어서 settings를 development로 해도 base.py의 설정은 모두 적용되고, DB만 development.py를 따르는 것이다.

하지만 매번 환경변수를 입력하기 귀찮기때문에 DJANGO_SETTINGS_MODULE을 developement로 설정하는 방법이 있었다. manage.py를 까봐도 os.environ.setdefault('DJANGO_SETTINGS_MODULE', 'blog.config.base')로 수정한 내역이 있다.

위와같이 입력해주면 manage.py를 실행할 때 환경변수를 일일히 입력해주지 않아도 된다.

…

생각해보니 애초에 manage.py의 ‘blog.settings’를 ‘blog.config.base’로 변경했었다. 첨부터 ‘blog.config.development’로 변경했으면 되지 않았나 하는 생각도 든다.

그리고 이걸 다 검색으로만 알아냈는데… Tow Scoops of Django의 5장이 내가 삽질했던 내용이었다. 또한 5장에선 manage.py의 환경변수뿐 아니라 비밀키 등 셋팅에 필요한 다양한 내용을 담고 있었다. 목표는 TDD의 연습이었기 때문에 이건 일단 넘어가고 나중에 더 읽고 적용해보기로 했다.

git

사실 git을 이용해 버전관리 하면서 더 고생을 많이했다.

아니 그냥 놓친게 있으면 새 커밋을 했으면 괜찮았을텐데 굳이 한 커밋안에 settings 관련 내용을 다 넣고 싶어서 reset를 한 것이 스노우볼이 되어 커밋이 모두 꼬이고 잘 쓰지도 못하는 브랜치를 이용해 병합하려고 하다가 브랜치를 세 개인가 더 만들고 HEAD도 왔다갔다하고…

어찌어찌 해결은 했다. 해결한 방법도 제대로 알고 해결한게 아니고 ubuntu에서 쓸수있는 gui 깃 클라이언트 smartgit을 이용해서 main으로 간신히 병합했다.

해결하고보니 너무 한심했다. 이전 회사에서 계속 혼자만 작업해서 git을 그냥 저장용으로만 쓰고 그나마 git desktop에서 브랜치 한 두번 만들어서 쓴 정도이긴 했지만, 개발자가 git을 제대로 못써서 이렇게 고생하는게 말도 안된다는 생각이 들었다.

여튼 오늘 어떻게 해결했는지도 모르겠지만, 내일부턴 매일 git을 조금씩이라도 공부해야겠다.
