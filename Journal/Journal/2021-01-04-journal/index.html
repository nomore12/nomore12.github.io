aws pem 키 문제

aws 인스턴스를 만들면서 발급받는 pem키가 안먹는 문제가 발생했다. 이 블로그를 참고하여 해결했다.

connection refused

ec2 서버에서 가상환경으로 python3 manage.py runserver를 실행했으나 접속이 안됨. url을 통해서도 접속이 안되고, ip를 이용해도 접속이 안됨.

netstat을 이용해서 포트가 열리긴 했나 확인해봤는데 잘 듣고있음.

/var/log/amazon/ssm에 amazon-ssm-agent.log 파일이 있어서 보니 이런 내용이 나옴.

어쨌든 접속이 안되고 있었다. aws 보안그룹은 제대로 지정해놨고, netstat으로 22번 ssh포트도 잘 되고있으니 8000번 포트가 안될리 없는데 왜 안되는지 모르겠다. 어쨌든 connection refused가 나고있으니 일단 nginx를 설치하고 8000번 포트를 nginx에서 강제로 열어봤다. /etc/nginx/conf.d/defaul.conf에서 Listen 8000;을 추가했다. 일단 8000번으로 들어가니 Welcome Nginx는 나온다. 잘됐다 싶어서 runserver를 하니 8000번 포트를 이미 사용하고 있어서 안된단다. 그리고 생각해보니 전에 runserver를 하면 127.0.0.1:8000으로 서버가 떴었다. 127.0.0.1은 로컬 호스트니까 밖에선 못들어오는게 아닐까?

그리고 클린 코드를 위한 테스트 주도개발 개정판 웹페이지에 들어가보니 runserver를 0.0.0.0:8000으로 열고있었다. 그래서 0.0.0.0:8000로 runserver를 해보니 이번에도 8000번 포트를 쓰고있어서 안열린다고 한다. nginx의 default.conf에 추가한 Listen 8000;을 삭제하고 다시 runserver를 하니 일단 “DissllowedHost at /“이라는 장고 에러 페이지는 열렸다. 사실 nginx도 설치하지 않아도 되었던 것이었다.

나는 왜 어제 하루종일 ec2와 씨름했을까… 인스턴스를 몇 번이나 지웠다 만들고, 호스팅도 몇 번이나 다시하고, 온갖 방법을 다 써봤는데 안되서 일단 자고 내일하자 했는데… 일어나서 잠깐 해보니 한 번에 됐다.

다들 개발자는 끈질기게 붙들고 어떻게든 해결해야 된다고 해서 자면서 너무 찝찝했었는데, 이번에 머리를 좀 환기시키고 난 후 보니 한 번에 해결이 되었다. 그리고 문제도 간단하게 해결 할 수도 있었는데, 한 곳에만 몰두하다 보니 다른 방법이 보이질 않아서 그랬던 것 같다. 일단 문제가 생겼을 때는 끈질기게 파고 들다가도 잠깐씩 머리를 쉬었다가 다시 해보는 것도 좋겠다는 생각을 했다.

gunicorn

WSGI는 웹 어플리케이션과 웹 서버가 통신하기 위한 인터페이스. 프로토콜 개념으로 이해할 수도 있다. WSGI는 서버와 앱 양단으로 나뉘어 있음. WSGI 리퀘스트를 처리하려면 서버에서 환경정보와 콜백함수를 앱에 제공해야 함. 앱은 그 요청을 처리하고 콜백함수를 통해서 서버에 응답한다.

응 몰라.

위키백과를 보면

  • 환경변수가 바뀌면 타겟 URL에 따라서 리퀘스트의 경로를 지정해준다.
  • 같은 프로세스에서 여러 어플리케이션과 프레임워크가 실행되게 한다.
  • XSLT 스타일시트를 적용하는 것과 같이 전처리를 한다.

WSGI에 대한 설명은 여기에 아주 잘 나와있다^^

gunicorn은 힘쎄고 빠른 WSGI라고 한다. 일단 책에서 까라니까 까기로 한다.

기능 테스트를 해보면 css를 적용하지 않았을 때 처럼 AssertionError: 100.5 != 512 within 10 delta (411.5 difference)가 발생한다. runserver는 Django 개발 서버가 정적파일(css등)을 알아서 제공해주지만, 우리의 귀여운 녹색 유니콘은 그걸 안한다. 게을러 빠진새끼. 그래서 nginx에 대신하도록 설정해주어야 한다.

일단 pip로 gunicorn을 설치하고, gunicorn으로 웹 서버를 띄워본다.

일단 서버는 잘 뜬것같다. 근데 사이트에 접속해보면 css가 적용이 안되어 있다. 그리고 기능 테스트를 하면 무언가 잘못되었다는 것을 알 수 있다.

아이템을 추가하는 테스트는 통과하지만, css 테스트는 통과하지 못했다. django의 runserver는 정적 파일을 알아서 처리해주지만 gunicorn은 그렇지 못하기 때문이다. 그래서 nginx가 이것을 대신하도록 해주어야 한다.

/etc/nginx/sites-available/sitename.conf를 수정해준다.

아래쪽에 nginx가 스태틱 폴더를 찾아가도록 설정했다. 하지만 책이랑 웹 염소는…

으로 되어있었다. 개발자 도구에서도 css를 못받고 있었다. nginx error.log를 확인해보니 css가 퍼미션이 없어서 접근이 안되는데

요렇게 나온다. 책이랑 웹 염소때는 nginx가 무언가 변했거나, nginx 책보고 이건 root로 해야되는게 아닌가 해서 바꿨는데 이것 때문에 그런걸수도 있다. 그리고 원래는 프로젝트를 /root/에 넣어놨었는데, 어떤 스택 오브 플로우에선 프로젝트를 /root/에 넣으면 안된다고 해서 /home/폴더로 빼긴 했었다. 여튼, 결정적으로 동작하게 된 계기는 nginx의 location에서 static을 뺀 것이었다. nginx의 root와 alias는 나중에 정리 할 필요가 있을 것 같다.

root 폴더에 있어서 스태틱 파일에 제대로 접근하지 못했던 것 같다. 이후 예제에서 alias로 static경로 넣어도 잘 된다.
