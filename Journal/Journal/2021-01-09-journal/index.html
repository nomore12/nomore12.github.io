postgresql

Django에서 Postgresql이 좋다길래 테스트를 해봤다. “PostgreSQL로 시작하는 SQL코딩 입문 기본편, 활용편”을 도서관에서 빌려서 기본편을 보면서 했다. 일단 책 제목은 “SQL 코딩입문”인데 절대 SQL 코딩 입문이 아니다. db를 아주 잘쓴다고는 못해도 검색해가면서 어찌어찌 sql문도 만들고, django 쓰면서 orm도 써보고 해서 그나마 보는데 많이 어렵지는 않았는데(기본편 한정) 입문자가 보면 포기 할 것 같다.

어쨌든, postgresql 커맨드라인에서 연습용 DB를 만들고 pdAdmin4와 Dbeaver로 연결해봤는데 MySql과 뭔가 좀 다르다. 나는 분명 Database를 만들었다고 생각했는데 이름만 데이터베이스고 연습용 데이터베이스 안의 스키마가 데이터베이스 역할을 하고 있었다. 스키마에서 테이블을 만들고 컬럼을 정의하고 해당 스키마에서 sql 질의를 작성해야한다.

그리고 public 스키마가 자동 생성되어 있다. public 스키마는 테이블이 없는 상태이고, sql기본 질의도 public으로 잡혀있다. study 스키마를 만들어서 study의 student를 가져오려면

위의 예시처럼 스키마 이름을 입력하고 테이블을 명시해야 한다. 뭔가… 익숙한 구조는 아니다.

Postgresql의 구성요소는 클러스터 -> 데이터베이스 -> 스키마 -> 테이블의 순서로 보면 된다.

클러스터

클러스터는 실행중인 데이터베이스 서버의 단일 인스턴스에 의해 관리되는 데이터베이스의 집합으로, 데이터베이스 클러스터 또는 카탈로그 클러스터라 한다.

클러스터는 이름을 식별되지 않으며 Postgresql 내에서 클러스터를 참조할 수 있는 방법은 없다(이게 무슨말인지 모르겠었는데 데이버베이스 설명을 보니 알겠더라)

데이터베이스

데이터베이스는 스키마의 집합이다.

데이터베이스의 명칭은 클러스터 내에서 유일해야 한다.

하나의 데이터베이스 내에서는 다른 스키마의 테이블들과 조인이 가능하다.

이렇게 보니 뭔가 이해가 간다. 테이블의 집합이 스키마(일반적으로 데이터베이스라고 표현하는) 스키마의 집합이 데이터베이스이다. 하나의 데이터베이스에 있는 스키마들은 각각의 테이블 데이터를 join이 가능한 것이다. 규모가 큰 서비스에서 비지니스 로직, 서비스 로직 등을 분리하기 위해 스키마를 나누고 필요할 경우 데이터를 조인해서 사용할 수 있을까 싶었다.

cli

mysql이랑 많이 달랐다. use database;라던가 show database; 같은 cli 명령어가 \c, \l 과 같은 이스케이프 + 영문자 조합으로 쓰였다. 이스케이프가 들어가지 않는 명령어도 있었지만. 여튼, 당장은 익숙하지 않아서 cli로 써야할 때는 검색해가면서 사용하고, 왠만하면 Dbeaver나 PdAdmin을 사용할 듯 하다.

Docker

인프런의 따라하며 배우는 도커와 CI환경을 수강하고 있다.

회사에서 급하니까 검색해가면서 그때그때 Dockerfile이나 docker-compose파일 수정하면서 썼었는데, 그러다보니 볼륨도 계속 꼬이고 nginx 설정에서도 계속 삽질하고 문제가 많았었다. 도커 도큐먼트를 봐도 잘 안됐다. Django나 Django Restframework 도큐먼트는 잘 읽혔는데 도커는 이상하게 잘 안읽혔다(그땐 리눅스도 잘 모르고 도커가 어떻게 돌아가는지도 모르고 그냥 써서 더 그랬을지도). 손목이 좀 나은후에는 도서관에서 책도 좀 빌려봤지만 뭔가 잘 안되서 도커는 항상 큰 산처럼 느껴졌었다.

근데 강의가 좋아서 그런건지 아니면 내가 책을 잘 못읽어서 그런지 이해가 잘 된다. 그제랑 오늘 반쯤 봤으니 다음주 초 정도까지 다 보고나서 다음 프로젝트들은 모두 도커로 해야것다.

도커 정리는 나중에 포스팅으로 올려야것다. 여기에 쓰려니 너무 많고, 앞으론 일지에는 구체적인 사용법이나 메뉴얼, 트러블슈팅 등은 포스팅 안할련다.
