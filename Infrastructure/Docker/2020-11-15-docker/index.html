도커 사용의의

도커의 사용의의를 내가 납득하기 위해 ‘도커/쿠버네티스를 활용한 컨테이너 개발 실전 입문’를 참고하여 간단하게 요약함.

환경차이로 인한 문제 방지

  • 로컬 개발 환경에서 필요한 애플리케이션을 신속하게 갖출 수 있고, 플랫폼에 상관없이 배포 가능.
  • 프론트엔드에 apache나 nginx등의 웹 서버를 두는것도 간단하게 해결 가능.
  • 도커를 사용하면 코드로 관리하는 인프라와 불편 인프라의 두 개념을 간단하고 낮은 비용으로 실현 할 수 있음
    • 코드 인프라
      • 코드 기반으로 인프라르르 정의한다는 개념. 서버를 어떻게 구성할 것인지, 어떤 라이브러리와 도구를 설치할지를 코드로 정의하고 Chef나 Ansible같은 프로비저닝 도구로 서버를 구축.
      • $ nodebrea install-binary stable 과 같은 코드를 실행한 경우에 stable이 가리키는 버전은 생각보다 자주 변하기 때문에 항상 같은 결과가 보장되지 않음.
    • 코드 인프라에 대한 대책이 불편 인프라.
      • 불변 인프라는 어떤 시점의 서버상태(서버 이미지라고도 함)를 저장해 복제할 수 있게 하자는 개념. 제대로 설정된 상태의 서버를 항상 사용할 수 있다는 점이 가장 큰 장점.
      • 서버 변경이 필요한 경우, 기존 인프라를 수정하는 대신 새로운 서버를 구축하고 그 상태를 이미지로 저장한 다음 그 이미지를 복제한다. 한번 설정된 서버는 수정 없이 파기되므로 멱등성을 신경쓸 필요조차 없다.
  • 인프라 구성이 Dockerfile로 관리되므로 코드로 관리하는 인프라는 도커의 대원칙이다. 도커는 컨테이너형 가상화 기술을 사용한다. 컨테이너형 가상화는 운영체제 대부분을 호스트 운영체제와 공유한다. 그만큼 실행에 걸리는 시간이 짧아져 구성을 수정하지 않고 인프라를 완전히 새로 만드는 불변 인프라와 궁합이 잘 맞는다.

멱등성.
멱등법칙 또는 멱등성은 수학이나 전산학에서 연산의 한 성질을 나타내는 것으로, 연산을 여러 번 적용하더라도 결과가 달라지니 않는 성질을 의미한다. 멱등법칙의 개념은 추상대수학과 함수형 프로그래밍(참조 투명성의 성질과 관련된)의 여러 부분에서 사용되고 있다.


애플리케이션 구성 관리의 용이성

  • 도커 컨테이너는 애플리케이션 인프라를 함께 담은 상자다. 일정 규모를 넘는 시스템은 여러개의 컨테이너를 조합하지 않으면 시스템을 구성할 수 없음. 배포작업이 쉬워지긴 했지만, 각 컨테이너의 의존관계나 실행 순서가 어긋나면 제대로 동작하지 않을 수 있음.
  • 도커 컨테이너 오케스트레이션 시스템
    • 도커는 여러 컨테이너를 쉽게 관리할 수 있도록 Docker Compose라는 도구 제공. 도커 컴포즈를 사용하면 yaml포맷으로 작성된 설정 파일로 컨테이너를 정의하거나 컨테이너 간의 의존 관계를 정의해 시작 순서를 제어할 수 있다.
    • 대규모 트래픽 처리 같은 많은 작업을 해야하는 시스템은 도커(도커 노드)가 설치된 서버를 여러대 갖추고 이 노드 그룹에 필요한 만큼의 애플리케이션 컨테이너를 배포해야 함.
    • 도커 컴포즈가 단일 서버를 넘어 여러 서버에 걸쳐 있는 여러 컨테이너를 관리할 수 있도록 한 도구가 도커 스웜(Docker Swarm). 여러 컨테이너를 관리하는 것만이 목적인 도커 컴포즈와 달리 도커 스웜은 컨테이너 증가 혹은 감소는 물론이고 노드의 리소스를 효율적으로 활용하기 위한 컨테이너 배치 및 로드 밸런싱이 가능함.
    • 또한, 배포 시에도 롤링 업데이트(오래된 컨테이너와 새로운 컨테이너를 단계적으로 서비스에 교체 투입하는것)가 가능해 운여연에서도 장점이 많음. 이러게 여러 서버에 걸쳐 있는 여러 컨테이너를 관리하는 기업을 컨테이너 오케스트레이션 이라고 함.
